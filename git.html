Om GitHub

Git og GitHub er verktøy som ofte brukes for samarbeid mellom utviklere. Git lar utviklere spore endringer, arbeide parallelt på ulike deler av et prosjekt og håndtere forskjellige versjoner av koden (Chacon & Straub, 2014, s. 10–15). GitHub, en plattform som bygger på Git, lar brukere lagre og administrere prosjekter. GitHub fungerer også som et samarbeidsverktøy, der brukere kan bruke pull requests, issues og diskusjoner for å lette kommunikasjon og samarbeid (GitHub Guides, n.d.).

Fordeler med GitHub

Samarbeid mellom utviklere: I GitHub kan du samarbeide med andre utviklere på et prosjekt. Du kan også bruke denne plattformen for å dele koder med andre (Chacon & Straub, 2014, s. 10–15).

Mulig å jobbe lokalt på pc-en: GitHub lar hver utvikler ha en fullstendig kopi av prosjektet på sin lokale maskin. Dette gjør det mulig å arbeide offline og gir fleksibilitet i tilfelle problemer med internettforbindelsen (Chacon & Straub, 2014, s.15-16).

Versjonskontroll: GitHub lagrer alle versjoner av et prosjekt, slik at utviklere kan gå tilbake til tidligere stadier ved behov. Dette forhindrer tap av data og gjør feilsøking enklere (Chacon & Straub, 2014, s.10-13).

Gren: Grener skiller ulike arbeidsstrømmer i et repository. Endringer gjøres i egne grener for å holde dem isolert og enklere å kontrollere endringer. Dette gjør det mulig å jobbe med flere grener samtidig, der hver gren representerer et eget prosjekt eller funksjon. Et repository har en standardgren som fungerer som prosjektets gjeldende versjon, samt én eller flere arbeidsgrener med pågående endringer. Det anbefales å opprette en egen arbeidsgren i stedet for å endre direkte i standardgrenen (Carlyrevier et.al., 2024).

Commit forespørsler: Lar utviklere foreslå endringer, som teammedlemmer kan gjennomgå og diskutere før de blir en del av prosjektet (GitHub Docs, n.d.).

Utføring (merge): forespørselen din er godkjent blir endringene en del av standardgren. Hvis det er gjort endringer i samme kode, vil du få opp en melding og bli bedt om å løse denne konflikten før du kan utføre endringene (GitHub Docs, n.d.).

Pull forespørsler: Lar utviklere foreslå endringer, som teammedlemmer kan gjennomgå og diskutere før de blir en del av prosjektet (GitHub Docs, n.d.).

VS Code: En annen fordel med GitHub er at det enkelt kan integreres med VS Code. I VS Code kan du utføre alle handlingene som er nevnt ovenfor, samtidig som du får en oversiktlig visning av alle handlingene direkte i VS code.

Bruk av GitHub

Gruppen vår har brukt GitHub som hovedverktøy for samarbeid om eksamensoppgaven. Prosessen startet med at alle gruppemedlemmene opprettet hver sin GitHub-konto. Deretter opprettet vi et felles repository for prosjektet, som vi navnga "DATA1200" etter emnekoden. Dette repository ble koblet til VS Code sammen med GitHub-brukerne til alle i gruppen, slik at vi kunne arbeide effektivt og strukturert.

Vi startet med å opprette HTML-filer for de ulike sidene beskrevet i eksamensoppgaven, samt en tilhørende CSS-fil. Filene ble i hovedsak opprettet og redigert i VS Code. For å sikre en smidig arbeidsflyt fordelte vi arbeidsoppgaver og seksjoner blant gruppemedlemmene. Dette bidro til å redusere risikoen for konflikter i koden når vi skulle gjøre commits. Til tross for denne strategien, opplevde vi noen tekniske utfordringer i starten, spesielt med commit og pull requests. Dette skyldtes at vi ikke hadde konfigurert e-post og brukernavn korrekt i VS Code. Etter å ha løst dette problemet, kunne vi enkelt legge til og dele endringer.

I løpet av prosjektet støtte vi også på utfordringer som følge av at fem personer arbeidet med samme kode. Til tross for at vi prøvde å unngå det, oppstod det enkelte ganger konflikter når flere endret på samme del av koden. Dette krevde ekstra tid og innsats for å løse. I tillegg opplevde vi andre feilmeldinger som forsinket arbeidet, samt situasjoner der samtidige endringer fra flere utviklere gjorde det tidkrevende å commite og synkronisere.

Til tross for utfordringene lærte vi mye om samarbeid med GitHub og VS Code. Selv om vi fortsatt har en del å lære, har vi oppnådd en god forståelse for arbeidsflyten og verktøyene. Prosjektet har vært en verdifull erfaring som har styrket vår evne til å arbeide som et team i et teknisk miljø.

Vi inkludert to skjermbilder som viser commits utført av de ulike gruppemedlemmene i repositoryet, slik det vises i GitHub. Dette illustrerer bidragene fra hver enkelt i gruppen og gir en oversikt over vår felles innsats.

Ovenfor vises to skjermbilder av commits som er gjennomført i GitHub. I disse visningene kan man se meldingene som ble lagt ved hver commit for å forklare hvilke endringer som er gjort. Det vises også informasjon om tidspunktet for endringene og hvem som har utført dem.

Ved å klikke inn på commit-historikken får man en detaljert oversikt over hvilke endringer som er gjort i koden. Endringene presenteres visuelt, hvor koden som er fjernet, vises i rødt, mens koden som er lagt til, vises i grønt. I tillegg er det deler av koden som er sammenklemt og merket med blått. Disse delene kan åpnes for å se den uendrede koden og gi bedre kontekst til de utførte endringene.

Denne funksjonaliteten gir en tydelig og strukturert oversikt over endringer i prosjektet, noe som gjør det enklere å samarbeide og holde oversikt over hva som er blitt oppdatert.
